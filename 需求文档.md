# Git 提交影响评估 Agent 需求文档

## 1. 目标

读取指定 Git 仓库在配置范围内的提交，分析改动影响并生成可供研发快速阅读的 Markdown 报告。

## 2. 输入与范围

- `REPO_PATH`：待分析的本地 Git 仓库。
- `NUM`：默认分析最近多少次提交。
- `--limit`、日期和提交范围参数可以临时覆盖默认范围。
- 读取提交元数据、diff、变更文件上下文和 Git `numstat` 代码行统计；不修改目标仓库。
- `FEISHU_ENABLED=true` 且配置 `FEISHU_WEBHOOK_URL` 时，报告生成后向飞书机器人发送摘要和报告路径。
- 飞书摘要必须使用卡片消息，仅包含报告第 2、3、4 节的压缩内容：总体结论、主要功能改动和涉及模块。
- 摘要中的总体结论保留原意并压缩为短文本；功能改动和模块仅展示前若干项，同时标注原始总数。

## 3. 报告格式需求

报告必须按以下顺序输出：

1. **提交时间线与变更概览**：位于报告最前面。
   - 汇总分析提交数、提交人数、新增行数和删除行数。
   - 以 Markdown 表格展示最近的提交：提交时间、提交人、提交内容、涉及文件数、新增/删除行数。
   - 使用可读的横向字符条表示每次提交的相对代码变更量；条越长，表示新增与删除行数越多。
   - 当分析提交数超过 20 时，时间线只展示最近 20 次，其余提交仍参与整体分析。
2. **分析范围**：保留现有内容，说明仓库、分析条件和实际提交数。
3. **总体结论**：保留现有内容和总体风险等级。
4. **主要功能改动**：保留现有内容。
5. **涉及模块**：保留现有内容。
6. **潜在 bug 和风险**：列出风险与证据。
7. **建议人工重点检查的位置**：列出后续检查建议。
8. **重点提交明细**：必须位于报告最后。
   - 不逐条罗列所有提交。
   - 按风险等级和代码变更量筛选，最多展示 10 条。
   - 每条仅包含重点原因、提交人和时间、代码量、涉及文件、分析结论和主要风险。

## 4. 代码量统计规则

- 使用 Git `numstat` 统计新增行和删除行。
- 二进制文件不计入代码行数。
- 被项目忽略规则排除的锁文件、构建产物和压缩资源不计入代码行数。

## 5. 可靠性要求

- 模型返回必须通过 Pydantic 数据校验后才能用于报告。
- 网络错误、非 JSON 返回、字段缺失或字段类型不符合要求时，按配置重试。
- 默认最多重试 3 次，使用递增等待时间。
- 重试耗尽后降级为本地基础分析，并在运行日志中记录原因。
